  • a. {{note|...}}The current default format is binary.
  • b. {{note|...}}The "classic" format is plain text, and an XML format is also supported.
  • c. {{note|...}}Theoretically possible due to abstraction, but no implementation is included.
  • d. {{note|...}}The primary format is binary, but a text format is available.<ref>...</ref>
  • e. {{note|...}}Means that generic tools/libraries know how to encode, decode, and dereference a reference to another piece of data in the same document. A tool may require the IDL file, but no more. Excludes custom, non-standardized referencing techniques.
  • f. {{note|...}}ASN.1 does offer OIDs, a standard format for globally unique identifiers. However, there is no standard for "marking"/"tagging" an arbitrary piece of data in a document with an OID. There is also no standard format for locally unique identifiers within a document. Therefore, a generic ASN.1 tool/library can not automatically encode/decode/resolve references within a document without help from custom-written program code.

  • a. {{note|...}}One possible encoding; the specification document does not specifically give an encoding for this datatype.
  • b. {{note|...}}The RFC CSV specification only deals with delimiters, newlines, and quote characters; it does not directly deal with serializing programming data structures.
  • c. {{note|...}}The netstrings specification only deals with nested byte strings; anything else is outside the scope of the specification.
  • d. {{note|...}}XML in and of itself is not a data serialization language, but many data serialization formats have been derived from it; as such, there are many different ways, in addition to those shown, to serialize programming data structures into XML.
  • e. {{note|...}}This syntax is not compatible with the Internet-Draft, but is used by some dialects of Lisp.
